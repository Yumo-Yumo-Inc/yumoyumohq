# 匿名化彙總與 B2B 資料產品

## 5.8 匿名化彙總與 B2B 資料產品

這是產生 B2B 收益並在嚴格隱私保證下運作的介面。

### 轉換規則

從 `receipts + line_items + price_observations` 轉換為匿名化彙總是一個**單向、破壞性**的轉換：

1. **取代 user_id 與 wallet_address。** 使用與使用者識別碼分離的工作階段範圍雜湊。
2. **粗化地理資訊。** 僅限城市級粒度；座標若曾記錄則予以捨棄。
3. **粗化時間。** 高頻指數使用每日分桶；類別層級報告使用每週分桶；人口統計群組使用每月分桶。
4. **強制執行 k-匿名性。** 每筆發布的記錄必須屬於準識別欄位（`city`、`merchant_class`、`time_bucket`、`category_path`）上至少具有合約定義最小數量不同貢獻者的儲存格。低於門檻的儲存格被抑制或粗化直至通過。
5. **為計數型輸出添加校準的差分隱私雜訊。**
6. **從已驗證的 PoE 建立彙總範圍。** 訂單頁面收據與其他待驗證記錄保留於記憶層。

### 可用產品

| 產品 | 粒度 | 更新頻率 | 指示性價格 |
|---|---|---|---|
| **TR 通膨脈衝** | 類別 × 區域 × 週 | 每日 | $X / 月（訂閱） |
| **購物籃面板** | 標準商品 × 城市 × 週 | 每日 | $Y / 月 |
| **商家基準** | 連鎖 × 類別 × 月 | 每週 | $Z / 月 |
| **自訂群組查詢** | API 每次查詢，強制執行 k-匿名性門檻 | 隨選 | $Q / 查詢 |

價格與確切粒度為佔位符；生產型錄將於商業發布前定稿。

### B2B 回應範例

```json
// GET /b2b/v1/inflation-pulse?region=istanbul&category=food.dairy&from=2026-05-01&to=2026-05-17
{
  "region": "istanbul",
  "category": "food.dairy",
  "series": [
    { "week_start": "2026-05-04", "index": 100.0, "n_observations": "<n>", "n_distinct_contributors": "<n>" },
    { "week_start": "2026-05-11", "index": 101.7, "n_observations": "<n>", "n_distinct_contributors": "<n>" }
  ],
  "k_anonymity_floor_met": true,
  "methodology_version": "1.0.0"
}
```

每個 B2B 回應攜帶 `n_distinct_contributors`，使買方能審計 k-匿名性底線是否已達成。低於底線的儲存格以 `suppressed: true` 回傳，不含數值。具體門檻值由內部營運層管理。

### 超出範圍的欄位

- 任何綁定至單一使用者的欄位。
- 任何低於所查詢儲存格 k-匿名性門檻的記錄。
- 座標、地址、電話號碼。
- 付款工具中繼資料。
- 跨查詢可連結的匿名化 ID（每次查詢滾動新的工作階段雜湊）。

### 與現有面板的比較

Yumo Yumo 的 B2B 資料產品與 Nielsen、GfK、Kantar 及 SimilarWeb 爭奪相同的買方。相較於這些面板：

- **收據層級**，而非調查回憶 — 較少的測量誤差假影。
- **更高的更新頻率** — 每日對比每週/每月。
- **新興市場覆蓋** — 以土耳其為首；既有業者在土耳其的覆蓋最薄弱。
- **更低的每筆記錄成本** — 面板由既有使用者活動建構，而非付費受訪者。

取捨：Yumo Yumo 的面板於發布時規模較小，且偏向早期採用者人口統計。
