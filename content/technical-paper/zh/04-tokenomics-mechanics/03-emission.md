# 用户奖励发行

## 4.3 奖励如何流入 bINT

用户奖励池由 4.17 的分配表治理。在该池内部，每日发行由一条曲线计量，该曲线以月活跃使用量作为主要输入。这条曲线有三个值得点明的性质：

- **向峰值的阶梯式增长。** 随着 MAU 跨越既定的级距增长，每日发行池以阶梯方式而非连续方式扩张。这避免了活跃度在某一阈值附近振荡时的断崖效应。
- **有界的峰值。** 每日池阶梯式增长至一个峰值级距，随后保持。达到峰值后，额外的 MAU 会提高每用户的贡献密度。各级距数值在生产环境中校准，不予公布。
- **长周期。** 用户奖励轨道的规模设计为可维持 15 年的周期。供应量中的奖励份额（64.35 billion INT，见 4.17）是预算；曲线是计量器。

该阶梯函数 —— MAU 级距、各级距的每日池数值与过渡行为 —— 记录于 4.19。级距边界会随着观察到的活跃度演进而重新调校。

## 4.4 bINT → INT 转换生命周期

当一张收据通过信任层（03）时，bINT 在链下累积。它通过一个周期性的（每周）结算周期（epoch）结算为 INT，而非逐用户的链上转换调用。生命周期如下：

```
accrue  →  hold  →  settle (epoch)  →  claim  →  INT in user wallet
```

- **累积（Accrue）。** 按收据，在链下会计层进行。金额由信任级距、用户的每日上限与当前发行阶梯设定。
- **持有（Hold）。** bINT 在会计层中经过一个最短持有窗口后，才有资格结算。该窗口给信任层（03）时间，在任何 INT 被分发之前对异常行为作出响应。
- **结算（Settle）。** 每个 epoch，符合条件的 bINT 以固定 1:1 的比率（4.24）转换为 INT。引擎构建一份分发列表，由独立验证器对其进行核验（4.17），生成的根被发布至经审计的分发器。
- **领取（Claim）。** 用户直接从分发器将其 INT 领取到标准 SPL 钱包中，可转让。金库在领取前持有 INT；没有单独的归属步骤。

当某个 epoch 的合格奖励总额超过全局发行上限时，每位参与者的金额都以相同的系数按比例下调（软上限按比例分配，4.24）。持有窗口的长度与全局上限值在营运层管理，不予公布。

## 4.5 代币经济学意义上的每日上限

有效的每日 bINT 上限是基础上限、等级乘数（03 §3.6）与用户当前健康度（03 §3.5）的乘积。当前 MVP 实现使用按等级的表格（4.22）；目标架构使用基于公式的上限（4.23）。乘数与健康度数值为用户专属，存放于信任层。

这种分解很重要，因为它让协议可在保留代币经济学的前提下重新调校这三个因素中的任何一个。市场扩张可以提高基础值；等级体系再平衡可以调整乘数；滥用浪潮可以压缩健康度分布。
