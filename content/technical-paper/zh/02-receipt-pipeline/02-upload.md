# 階段 0 — 上傳

## 2.3 階段 0 — 上傳與預處理

### 客戶端

客戶端調整圖片尺寸並套用失真壓縮——在保持 OCR 所需解析度的同時，將一般手機的上傳大小控制在較小範圍。確切的解析度與品質目標由內部營運層管理。上傳前會移除 EXIF 中繼資料；地理資訊擴充僅在使用者明確選擇加入時啟動。

用於去重的感知雜湊（perceptual hash）在伺服器端於收到圖像後計算（2.3.3）。

### 伺服器端

```json
// UploadRequest
{
  "user_id": "uuid",
  "content_type": "image/jpeg",
  "size_bytes": 524288,
  "captured_at": "2026-05-17T14:23:00Z"
}

// UploadResponse
{
  "receipt_id": "uuid",
  "upload_url": "https://...",
  "expires_at": "2026-05-17T14:24:00Z"
}
```

伺服器依生產環境定義的限制驗證上傳大小，將內容型別列入允許清單（`image/jpeg`、`image/png`、`application/pdf`），並簽發短期預簽名 URL。PUT 成功後，客戶端呼叫 `POST /receipts/{id}/process` 進入階段 1。

### 去重

在任何昂貴作業開始前，系統會執行多訊號感知相似度檢查。兩種情況被區分開來：

1. **同使用者重複** — 同一使用者重複上傳同一張收據時，系統會解析至既有記錄。這可防止意外重複上傳。
2. **跨使用者碰撞** — 看似在多個帳號間共享的收據會被標記為信任審查（階段 6 會予以降級）。這是反耕作（anti-farming）防禦的一部分。

同使用者重複屬於**軟成功** — 使用者會看到先前的結果。跨使用者訊號仍會繼續通過管線；信任評分器做出最終決定。確切的相似度門檻與訊號在生產環境中調校，並由內部營運層管理。
