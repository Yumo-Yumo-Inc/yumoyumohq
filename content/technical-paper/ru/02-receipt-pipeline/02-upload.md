# Этап 0 — Загрузка

## 2.3 Этап 0 — Загрузка и предобработка

### Сторона клиента

Клиент изменяет размер изображения и применяет сжатие с потерями — достаточное для OCR, но при этом сохраняющее небольшой размер загрузки на типичных телефонах. Точные целевые значения разрешения и качества управляются внутренним операционным слоем. Метаданные EXIF удаляются перед загрузкой; гео-обогащение начинается только при явном согласии пользователя.

Перцептивное хэширование для дедупликации выполняется на стороне сервера после получения изображения (2.3.3).

### Сторона сервера

```json
// UploadRequest
{
  "user_id": "uuid",
  "content_type": "image/jpeg",
  "size_bytes": 524288,
  "captured_at": "2026-05-17T14:23:00Z"
}

// UploadResponse
{
  "receipt_id": "uuid",
  "upload_url": "https://...",
  "expires_at": "2026-05-17T14:24:00Z"
}
```

Сервер проверяет размер загрузки на соответствие ограничению, определённому в продакшене, заносит тип контента в белый список (`image/jpeg`, `image/png`, `application/pdf`) и выдаёт краткосрочный предподписанный URL. После успешного PUT клиент вызывает `POST /receipts/{id}/process` для перехода на Этап 1.

### Дедупликация

Перед началом любой ресурсоёмкой работы выполняется многосигнальная проверка перцептивного сходства. Различаются два случая:

1. **Дубль того же пользователя** — повторная загрузка одного и того же чека тем же пользователем разрешается в существующую запись. Это предотвращает случайные повторные загрузки.
2. **Межпользовательское столкновение** — чеки, которые, по-видимому, используются совместно между аккаунтами, помечаются для проверки доверия (Этап 6 понижает их оценку). Это часть защиты от фарминга.

Дубль того же пользователя является **мягким успехом** — пользователь видит свой предыдущий результат. Межпользовательский сигнал всё равно проходит через конвейер; решение принимает скорер доверия. Точные пороги сходства и сигналы настраиваются в продакшене и управляются внутренним операционным слоем.
