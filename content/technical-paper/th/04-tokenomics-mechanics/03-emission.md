# การปล่อยรางวัลผู้ใช้

## 4.3 รางวัลไหลเข้า bINT อย่างไร

พูลรางวัลผู้ใช้ถูกกำกับโดยตารางการจัดสรรใน 4.17 ภายในพูลนี้ การปล่อยรายวันจะถูกวัดด้วยเส้นโค้งที่ใช้จำนวนผู้ใช้งานต่อเดือน (MAU) เป็นข้อมูลนำเข้าหลัก เส้นโค้งนี้มีคุณสมบัติสำคัญ 3 ประการ:

- **เติบโตแบบขั้นบันไดไปสู่จุดสูงสุด** เมื่อ MAU เพิ่มผ่านแต่ละช่วงที่กำหนด พูลการปล่อยรายวันจะขยายเป็นขั้น ๆ แทนที่จะเพิ่มอย่างต่อเนื่อง วิธีนี้ช่วยหลีกเลี่ยงผลกระทบแบบหน้าผาเมื่อกิจกรรมแกว่งอยู่ใกล้เกณฑ์
- **จุดสูงสุดที่มีขอบเขต** พูลรายวันเติบโตแบบขั้นบันไดไปสู่ช่วงสูงสุด จากนั้นคงที่ หลังผ่านจุดสูงสุดแล้ว MAU ที่เพิ่มขึ้นจะเพิ่มความหนาแน่นของการมีส่วนร่วมต่อผู้ใช้ ค่าของช่วงถูกปรับเทียบในระบบผลิตและไม่เผยแพร่
- **ขอบเขตระยะยาว** ราง User Rewards มีขนาดรองรับขอบเขต 15 ปี ส่วนแบ่งรางวัลของอุปทาน (64.35 พันล้าน INT, ดู 4.17) คืองบประมาณ ส่วนเส้นโค้งทำหน้าที่เป็นตัววัด

ฟังก์ชันแบบขั้นบันได ได้แก่ ช่วงของ MAU ค่าพูลรายวันในแต่ละช่วง และพฤติกรรมระหว่างการเปลี่ยนผ่าน จัดทำเอกสารไว้ใน 4.19 ขอบเขตของช่วงจะถูกปรับจูนใหม่ตามกิจกรรมที่สังเกตได้

## 4.4 วงจรชีวิตการแปลง bINT → INT

bINT สะสมนอกบล็อกเชนทันทีที่ใบเสร็จผ่านชั้นความน่าเชื่อถือ (03) และชำระเป็น INT ผ่านรอบ (epoch) เป็นระยะ (รายสัปดาห์) แทนที่จะเป็นการเรียกแปลงบนบล็อกเชนต่อผู้ใช้ วงจรชีวิตมีดังนี้:

```
accrue  →  hold  →  settle (epoch)  →  claim  →  INT ในกระเป๋าเงินผู้ใช้
```

- **Accrue (สะสม)** ต่อหนึ่งใบเสร็จ ในชั้นบัญชีนอกบล็อกเชน จำนวนถูกกำหนดโดยช่วงความน่าเชื่อถือ เพดานรายวันของผู้ใช้ และขั้นการปล่อยในขณะนั้น
- **Hold (ถือครอง)** bINT คงอยู่ในชั้นบัญชีตลอดหน้าต่างการถือครองขั้นต่ำก่อนมีสิทธิ์ชำระ หน้าต่างนี้ให้เวลาชั้นความน่าเชื่อถือ (03) ตอบสนองต่อรูปแบบที่ผิดปกติก่อนที่ INT จะถูกแจกจ่าย
- **Settle (ชำระ)** ในแต่ละรอบ (epoch) bINT ที่มีสิทธิ์จะแปลงเป็น INT ตามอัตราส่วนคงที่ 1:1 (4.24) เครื่องยนต์สร้างรายการแจกจ่าย ผู้ตรวจสอบอิสระตรวจทาน (4.17) และรากที่ได้จะถูกเผยแพร่ไปยังตัวกระจาย (distributor) ที่ผ่านการตรวจสอบ
- **Claim (เคลม)** ผู้ใช้เคลม INT ของตนโดยตรงจากตัวกระจายเข้าสู่กระเป๋า SPL มาตรฐาน สามารถโอนได้ คลังสำรองถือ INT ไว้จนกว่าจะมีการเคลม ไม่มีขั้นตอน vesting แยกต่างหาก

เมื่อรางวัลที่มีสิทธิ์รวมในรอบหนึ่งเกินเพดานการปล่อยระดับสากล จำนวนของผู้เข้าร่วมทุกคนจะถูกปรับลดด้วยตัวคูณเดียวกัน (soft-cap pro-rata, 4.24) ความยาวของหน้าต่างการถือครองและค่าเพดานระดับสากลถูกจัดการในชั้นดำเนินงานและไม่เผยแพร่

## 4.5 เพดานรายวัน ในแง่ tokenomics

เพดาน bINT รายวันที่มีผลจริง คือผลคูณของเพดานฐาน ตัวคูณระดับ (03 §3.6) และสถานะสุขภาพปัจจุบันของผู้ใช้ (03 §3.5) การใช้งาน MVP ปัจจุบันใช้ตารางตามระดับ (4.22) สถาปัตยกรรมเป้าหมายใช้เพดานแบบสูตร (4.23) ส่วนตัวคูณและค่าสุขภาพเป็นค่าเฉพาะของผู้ใช้และอยู่ในชั้นความน่าเชื่อถือ

การแยกส่วนประกอบเช่นนี้สำคัญ เพราะเปิดให้โปรโตคอลสามารถปรับจูนปัจจัยใดปัจจัยหนึ่งจากสามปัจจัยนี้ได้ โดยยังคงรักษา tokenomics โดยรวมไว้ การขยายตัวของตลาดอาจยกเพดานฐานขึ้น การปรับสมดุลของระบบระดับอาจเปลี่ยนตัวคูณ และคลื่นของความพยายามฉวยโอกาสอาจกดการกระจายของค่าด้านสุขภาพให้แคบลง
