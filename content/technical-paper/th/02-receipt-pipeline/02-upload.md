# ขั้นตอนที่ 0 — การอัปโหลด

## 2.3 ขั้นตอนที่ 0 — การอัปโหลดและการประมวลผลเบื้องต้น

### ฝั่งไคลเอนต์

ไคลเอนต์ปรับขนาดภาพและใช้การบีบอัดแบบสูญเสีย — ความละเอียดเพียงพอสำหรับ OCR ในขณะที่ยังคงรักษาขนาดไฟล์อัปโหลดให้เล็กบนโทรศัพท์ทั่วไป เป้าหมายความละเอียดและคุณภาพที่แน่นอนถูกจัดการในชั้นปฏิบัติการภายใน ข้อมูลเมตา EXIF ถูกลบออกก่อนอัปโหลด การเติมข้อมูลทางภูมิศาสตร์เริ่มต้นจากการยินยอมโดยชัดแจ้งของผู้ใช้

การแฮชเชิงการรับรู้ (perceptual hash) สำหรับการตรวจสอบซ้ำจะดำเนินการฝั่งเซิร์ฟเวอร์เมื่อได้รับภาพแล้ว (2.3.3)

### ฝั่งเซิร์ฟเวอร์

```json
// UploadRequest
{
  "user_id": "uuid",
  "content_type": "image/jpeg",
  "size_bytes": 524288,
  "captured_at": "2026-05-17T14:23:00Z"
}

// UploadResponse
{
  "receipt_id": "uuid",
  "upload_url": "https://...",
  "expires_at": "2026-05-17T14:24:00Z"
}
```

เซิร์ฟเวอร์ตรวจสอบขนาดการอัปโหลดตามขีดจำกัดที่กำหนดในระบบผลิต อนุญาตประเภทเนื้อหา (`image/jpeg`, `image/png`, `application/pdf`) และออก URL ที่ลงนามล่วงหน้าระยะเวลาสั้น หลังจาก PUT สำเร็จแล้ว ไคลเอนต์เรียก `POST /receipts/{id}/process` เพื่อเข้าสู่ขั้นตอนที่ 1

### การตรวจสอบซ้ำ

การตรวจสอบความคล้ายกันเชิงการรับรู้แบบหลายสัญญาณจะทำงานก่อนเริ่มงานที่มีต้นทุนสูงใด ๆ โดยแยกออกเป็นสองกรณี:

1. **ซ้ำโดยผู้ใช้รายเดียวกัน** — การอัปโหลดซ้ำของใบเสร็จเดียวกันโดยผู้ใช้รายเดียวกันจะชี้ไปยังบันทึกเดิม สิ่งนี้ป้องกันการอัปโหลดซ้ำโดยไม่ได้ตั้งใจ
2. **การชนกันข้ามผู้ใช้** — ใบเสร็จที่ดูเหมือนถูกแชร์ระหว่างบัญชีจะถูกระบุสำหรับการตรวจสอบความน่าเชื่อถือ (ขั้นตอนที่ 6 จะลดระดับ) นี่เป็นส่วนหนึ่งของการป้องกันการฟาร์ม

การซ้ำโดยผู้ใช้รายเดียวกันเป็น**ความสำเร็จแบบอ่อน** — ผู้ใช้จะเห็นผลลัพธ์ก่อนหน้าของพวกเขา สัญญาณข้ามผู้ใช้ยังคงดำเนินผ่านไพพ์ไลน์ต่อไป ตัวให้คะแนนความน่าเชื่อถือเป็นผู้ตัดสินใจ ค่าเกณฑ์ความคล้ายคลึงและสัญญาณที่แน่นอนถูกปรับแต่งในระบบผลิตและจัดการในชั้นปฏิบัติการภายใน
