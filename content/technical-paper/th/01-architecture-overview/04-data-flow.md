# กระแสข้อมูล: จากใบเสร็จสู่ bINT

## 1.3 กระแสข้อมูล: จากใบเสร็จสู่ bINT

การเดินทางของใบเสร็จประกอบด้วยสองเฟส

**เฟส A — ซิงโครนัส (ผู้ใช้กำลังรอ):**

1. ไคลเอนต์บีบอัดภาพ ลบ EXIF และขอ URL อัปโหลดที่ลงนามล่วงหน้า
2. ภาพลงสู่ที่เก็บอ็อบเจกต์ เซิร์ฟเวอร์ตรวจสอบดัชนีการลบข้อมูลซ้ำแบบ perceptual-hash
3. ชั้น OCR (02 ขั้นตอน 1) แยกข้อความและกรอบขอบเขต
4. LLM router (02 ขั้นตอน 2) แยก `ReceiptExtraction` JSON ที่มีโครงสร้าง
5. ชั้น regex/กฎ (02 ขั้นตอน 3) ปรับยอดรวมให้ตรงกันและตรวจสอบวันที่
6. canonical matcher (02 ขั้นตอน 4) แก้ไขแต่ละรายการเป็นค่า canonical product ID
7. merchant resolver (02 ขั้นตอน 5) แนบเอกลักษณ์ผู้ค้า
8. trust scorer (03) ส่งออกคะแนนความน่าเชื่อถือในช่วง [0, 1] และระบบแสดงตัวอย่างที่ตรวจสอบแล้วให้ผู้ใช้

**เฟส B — อะซิงโครนัส (การชำระเงินเบื้องหลัง):**

9. หากคะแนนความน่าเชื่อถือผ่านเกณฑ์ จะเขียนแถว `bINT.pending` ลงในบัญชีแยกประเภท
10. settlement worker รวมเครดิตที่รอดำเนินการเป็นชุดรายชั่วโมง คำนวณเพดานรายวัน (03) และ mint bINT บน Solana ไปยัง ATA ที่แช่แข็งของผู้ใช้
11. indexer รับเหตุการณ์ mint บนเชนและยืนยันกลับไปยังบัญชีแยกประเภทนอกเชน

ผู้ใช้เห็นเฟส A ภายในไม่กี่วินาที เฟส B สิ้นสุดโดยไม่มองเห็น สัญญาระหว่างสองเฟสคือ: **บัญชีแยกประเภทนอกเชนเป็นที่มาของความจริงจนกว่าจะมีการชำระเงินบนเชน** หลังจากนั้นสถานะบนเชนจะเป็นที่มาของความจริง และบัญชีแยกประเภทกลายเป็นกระจกอ่านแบบเร็ว
