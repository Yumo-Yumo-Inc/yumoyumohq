# ความขึ้นต่อกันและประเภทความล้มเหลว

## 1.9 ความขึ้นต่อกันและประเภทความล้มเหลว

เอกสารทางเทคนิคสาธารณะจัดประเภทความขึ้นต่อกันภายนอกตามพฤติกรรมของผลิตภัณฑ์ การกำหนดเส้นทางผู้ให้บริการ เกณฑ์ และการตอบสนองเหตุการณ์จัดการอยู่ในชั้นปฏิบัติการภายใน

| ประเภทความล้มเหลว | ผลกระทบต่อผู้ใช้ | ท่าทีสถาปัตยกรรมสาธารณะ |
|---|---|---|
| ความล่าช้าในการประมวลผลเอกสาร/AI | ตัวอย่างใบเสร็จล่าช้าหรือถูกส่งไปยังการตรวจสอบ | ขั้นตอน pipeline ถูกแยกโดยผลลัพธ์ที่มีประเภท แต่ละขั้นตอนสามารถเลื่อนออกไปหรือรันใหม่ได้อย่างอิสระ |
| ความไม่พร้อมใช้งานของระนาบข้อมูล | การประมวลผลใบเสร็จใหม่อาจหยุดชะงัก พฤติกรรมการอ่านอาจถูกจำกัด | โมเดลเหตุการณ์บัญชีแยกประเภทเป็น append-only การชำระเงินสามารถสร้างขึ้นใหม่ได้จากเหตุการณ์ที่เล่นซ้ำได้ |
| ปัญหาความพร้อมใช้งานของเชน/RPC | การชำระเงินบนเชนล่าช้า ตัวอย่างผู้ใช้ยังคงอยู่นอกเชนได้ | การบัญชีรางวัลเขียนลงในบัญชีแยกประเภทนอกเชนก่อน การเขียนบนเชนถือเป็นชำระเงินแบบรวมชุด |
| ความเสี่ยงด้านอำนาจและการลงนาม | อำนาจคลังหรืออำนาจโปรแกรมอาจได้รับผลกระทบ | อำนาจถูกแยกระหว่างกระเป๋าเงินของผู้ใช้ บริการแอปพลิเคชัน และชั้นบนเชน |
| การเติบโตของคิวและงานค้าง | เวลาการประมวลผลเพิ่มขึ้น งานที่มีลำดับความสำคัญต่ำถูกเลื่อนออกไป | กระแสผู้ใช้แบบซิงโครนัสและกระแสการชำระเงินเบื้องหลังแยกจากกัน |

การกำหนดเส้นทางผู้ให้บริการ เกณฑ์การลงนาม ขั้นตอนการกู้คืน และเวลาการตอบสนองเหตุการณ์จัดการอยู่ในชั้นปฏิบัติการภายใน เอกสารสาธารณะอธิบายผลกระทบต่อผลิตภัณฑ์และประเภทความยืดหยุ่นทางสถาปัตยกรรม

## อ้างอิงข้ามส่วน

- รายละเอียดการประมวลผลใบเสร็จ → 02 ระบบประมวลผลใบเสร็จรับเงิน
- แบบจำลองความน่าเชื่อถือและการป้องกันการใช้งานที่ไม่เหมาะสม → 03 ชั้นความน่าเชื่อถือ
- พื้นผิวโทเค็นและการชำระเงิน → 04 กลไก Tokenomics
- รูปแบบข้อมูลและผลิตภัณฑ์ข้อมูลรวม → 05 รูปแบบข้อมูลและ API
